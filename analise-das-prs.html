01 — Contexto do Desafio

O Challenge 003 pedia um Lead Scorer para o pipeline B2B da G4 Educação: um dataset com ~8.800 deals históricos e 2.089 deals ativos. A entrega esperada era uma ferramenta funcional que priorizasse o pipeline — não um relatório, não um notebook. A avaliação considerou qualidade do scoring engine, uso real de IA, evidências de processo e comunicação.

Critério central identificado

O avaliador avaliou sistematicamente três eixos: (1) o scoring engine está correto e usa os dados reais? (2) o processo com IA está documentado com evidências auditáveis? (3) a comunicação é clara o suficiente para um avaliador entender sem rodar o código?

04 — O Que o Avaliador Mais Valorizou

Evidência de raciocínio antes da execução

O feedback mais positivo foi para candidatos que mostraram análise manual dos dados antes de escrever uma linha de código. Felipe Calgaro fez tabela de critérios com pesos e pseudocódigo por fator. Lucas de Paula identificou 4 padrões no dataset que se tornaram os 4 componentes do score. Isso diferencia raciocínio próprio de delegação cega para a IA.

Documentação de falhas com causa raiz

Mateus Nogueira documentou 3 sessões de debug com causa raiz por bug — normalização de strings (GTXPro vs "GTX Pro"), NaN guards, conflito de prioridade de alertas, mutação de cache. O avaliador usou exatamente isso como evidência de "iteração real". Erros documentados corretamente valem mais do que código que funciona sem explicação.

Scoring calibrado nos dados, não em intuição

Lucas de Paula apresentou o funil matemático fechando (2.092 deals, soma dos KPI cards = 2.092). Bruno Campara calibrou o stagnation penalty no P75 histórico, não em valor arbitrário. Eduardo identificou e corrigiu o NULL em close_value para deals ativos. Cada um desses é evidência de que o candidato realmente rodou o código contra os dados reais.

Produto orientado à ação do vendedor

A visão Monday Morning (Mateus Nogueira), o Scoring Coach (Bruno Campara), a Agenda de Impacto e o Daily Focus Card foram explicitamente elogiados. O avaliador valorizou quem interpretou o problema como "ferramenta do vendedor" e não como "dashboard para o gestor".

05 — Critério Crítico: Evidências Visuais do Processo com IA

Padrão mais citado pelo avaliador

Em múltiplos PRs o avaliador pediu explicitamente evidências visuais do processo — não só log textual. Screenshots das conversas com IA e chat exports foram os formatos mais citados como preferidos. Esse foi o único ponto de melhoria pedido na submissão com nota 7.9 (PR #25 — Lucas de Paula). O process log em .docx foi explicitamente rejeitado como não auditável.

O que o avaliador considera evidência auditável:

  • ✓ Screenshots das conversas com IA (Claude Code, Cursor, etc.)
  • ✓ Chat exports em formato legível no repositório
  • ✓ Git history com commits descritivos incluindo contexto de decisão
  • ✓ DEVLOG cronológico com raciocínio das decisões (não só ações)
  • ✓ Testes que documentam comportamento esperado e casos de borda
  • ✗ Arquivo .docx com log (não renderizável no GitHub)
  • ✗ Pasta de screenshots criada mas vazia
  • ✗ Link para Google Drive com evidências (fora do repositório)

07 — Implicações para Sua Submissão

Com base no padrão de avaliação identificado, estes são os pontos que podem diferenciar ou eliminar sua submissão:

Obrigatório (eliminatório se ausente)

  • ✓ Evidências visuais do processo com IA — screenshots ou chat exports no repositório, não .docx
  • ✓ Scoring engine que demonstravelmente usa os dados reais do dataset
  • ✓ README com fórmula/pesos do score e como rodar
  • ✓ Nenhuma chave de API hardcoded no repositório

Diferenciadores (o que separa 7.x de 9.x)

  • ✓ Análise exploratória documentada dos dados ANTES do código — com achados numerados
  • ✓ Testes unitários para o scoring engine
  • ✓ DEVLOG cronológico com raciocínio de decisão (não só ações)
  • ✓ Sanity checks visíveis: distribuição do score por estágio, top 10 deals com breakdown
  • ✓ Fallback quando IA externa está indisponível
  • ✓ Commits descritivos incluindo contexto da decisão, não só a ação
Vantagem do seu perfil

Seu background como engenheiro de software + PM significa que você pode fazer o que a maioria não fez: raciocinar sobre os dados antes de delegar para a IA, depois documentar esse raciocínio de forma auditável. Os candidatos aprovados com as notas mais altas fizeram exatamente isso. Os reprovados ou revisados delegaram o raciocínio sem deixar rastro.
