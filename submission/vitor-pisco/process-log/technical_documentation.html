O Pipeline Coach AI é um assistente operacional de pipeline para times de vendas B2B. Ele transforma dados brutos de CRM em uma lista de 5 ações prioritárias por vendedor por dia — calculadas por um scoring engine baseado em regras, com explainability embutida.

Este documento cobre os três entregáveis obrigatórios: como rodar a solução, como o scoring funciona e por que foi construído assim, e o que o MVP ainda não faz com o caminho para escalar.

Três problemas conhecidos nos dados. Ignorá-los quebra os joins silenciosamente.

Problema 1 — Nome de produto inconsistente

O arquivo products.csv usa "GTX Pro" (com espaço). O arquivo sales_pipeline.csv usa "GTXPro" (sem espaço). A normalização é aplicada automaticamente ao carregar:

// Normalização aplicada no parse do products.csv
function normalizeProduct(name: string): string {
  return name.replace(/\s+/g, '')
  // "GTX Pro" → "GTXPro"  (único caso afetado)
}TypeScript

Problema 2 — Campo account vazio em 16,2% das linhas

1.425 das 8.800 linhas de sales_pipeline.csv têm account = "" (string vazia). Esses deals são válidos — apenas o link com a conta não foi registrado. A aplicação exibe "Conta não identificada" e nunca descarta essas linhas.

Problema 3 — close_value e close_date são strings vazias em deals abertos

Deals em Prospecting ou Engaging têm close_value = "" e close_date = "" — não são null, não são 0. Todo acesso a esses campos passa por:

function safeFloat(val: string, fallback = 0): number {
  if (!val || val.trim() === '') return fallback
  const n = parseFloat(val)
  return isNaN(n) ? fallback : n
}

02 — Scoring · 2.1
Visão Geral
do Scoring
Como cada deal aberto recebe uma pontuação de prioridade de 0 a 100.

Cada deal com deal_stage = "Prospecting" ou "Engaging" recebe um Priority Score de 0 a 100. Os 5 deals com maior score de cada vendedor se tornam as prioridades do dia.

O score é calculado inteiramente no browser, sem chamadas a servidor, usando os dados dos CSVs já carregados. O cálculo roda toda vez que o vendedor selecionado muda.

Data de referência fixa. O dataset é histórico (2016–2017). Para que os cálculos de "dias em aberto" façam sentido, toda operação de data usa REFERENCE_DATE = new Date('2017-12-27') como "hoje". Em produção com dados ao vivo, isso seria substituído por new Date().

Fórmula Resumida

PriorityScore = D1 (max 25) + D2 (max 25) + D3 (max 20) + D4 (max 20) + D5 (max 10)
              = máximo 100 pontosfórmula
02 — Scoring · 2.2
As 5
Dimensões
Cada dimensão pontua um aspecto diferente de urgência do deal.
D1 — Tempo sem Contato
25 pts
min(25, days_no_contact / 14 × 25)
Deals que não recebem atenção esfriaram. Threshold de 14 dias — após isso, pontuação máxima. No MVP, usa dias desde engage_date como proxy (tabela de interações não existe no dataset).
D2 — Aging da Oportunidade
25 pts
min(25, days_in_stage / team_avg × 25)
Deals que estão no mesmo stage por mais tempo que a média do escritório precisam de atenção. team_avg é calculado com todos os deals Won/Lost do mesmo regional_office.
D3 — Valor da Oportunidade
20 pts
min(20, est_value / portfolio_p90 × 20)
Deals maiores têm mais impacto se fechados. Comparação relativa ao P90 do portfólio do próprio rep — evita que um rep com deals pequenos nunca pontue nessa dimensão.
D4 — Stage Atual
20 pts
Engaging → 20 pts · Prospecting → 10 pts
Deals em Engaging estão em negociação ativa — têm maior probabilidade de fechar. Deals em Prospecting são prioridade menor. Won e Lost são excluídos do scoring.
D5 — Benchmark vs Equipe
10 pts
clamp(0, 10, (days_above_avg / 7) × 10)
Bônus para deals que estão significativamente acima da média do time — significa que já passaram do tempo esperado e correm risco real. Deals abaixo da média não recebem penalidade (score 0 nessa dimensão, não negativo).
02 — Scoring · 2.3
Fórmula
Completa
Implementação TypeScript exata usada no MVP.
const REFERENCE_DATE = new Date('2017-12-27')
const CONTACT_THRESHOLD_DAYS = 14

function calcPriorityScore(
  deal: Deal,
  teamAvgDays: number,        // média Won/Lost do mesmo escritório
  portfolioMaxValue: number   // P90 dos deals abertos do rep
): ScoreResult {

  const daysInStage = daysBetween(deal.engage_date, REFERENCE_DATE)

  // D1: Tempo sem contato (proxy: engage_date, sem tabela de interações)
  const d1 = Math.min(25, (daysInStage / CONTACT_THRESHOLD_DAYS) * 25)

  // D2: Aging relativo à média do escritório
  const avg = teamAvgDays > 0 ? teamAvgDays : daysInStage
  const d2 = avg > 0 ? Math.min(25, (daysInStage / avg) * 25) : 0

  // D3: Valor relativo ao portfólio do rep (P90)
  const p90 = portfolioMaxValue > 0 ? portfolioMaxValue : deal.est_value
  const d3 = p90 > 0 ? Math.min(20, (deal.est_value / p90) * 20) : 0

  // D4: Peso do stage
  const d4 = deal.deal_stage === 'Engaging' ? 20 : 10

  // D5: Bônus por estar acima da média da equipe
  const daysOver = Math.max(0, daysInStage - teamAvgDays)
  const d5 = Math.min(10, (daysOver / 7) * 10)

  const score = Math.round(d1 + d2 + d3 + d4 + d5)

  // Identificar driver principal (dimensão de maior contribuição)
  const drivers = [
    { score: d1, label: `${daysInStage}d sem contato` },
    { score: d2, label: `${daysInStage}d acima da média no stage` },
    { score: d3, label: 'maior valor no portfólio' },
    { score: d4, label: 'Engaging em andamento' },
    { score: d5, label: `${Math.round(daysOver)}d acima da média da equipe` },
  ]
  const primary = drivers.reduce((a, b) => a.score >= b.score ? a : b)

  return {
    score,
    label: score >= 85 ? 'Crítico' : score >= 70 ? 'Alto' : score >= 55 ? 'Médio' : 'Baixo',
    context_reason: `${primary.label} · ${deal.deal_stage} · $${(deal.est_value/1000).toFixed(1)}K`,
    breakdown: { d1, d2, d3, d4, d5 }
  }
}

function daysBetween(dateStr: string, reference: Date): number {
  if (!dateStr?.trim()) return 0
  const d = new Date(dateStr)
  return Math.round((reference.getTime() - d.getTime()) / 86_400_000)
}TypeScript
02 — Scoring · 2.4
Por Que
Esses Pesos
A lógica de negócio por trás de cada decisão de peso.

D1 + D2 dominam com 50% do score

Tempo e envelhecimento representam juntos 50 dos 100 pontos máximos. Essa escolha reflete o problema central que o produto resolve: deals bons esfriando por falta de contato. Um deal esquecido por 14 dias tem urgência objetiva independente do seu valor.

D3 vale 20% — valor importa, mas não domina

Valor está em segundo plano intencional. Um deal de $26.768 (GTK 500) não deve automaticamente tomar prioridade sobre um deal de $5.000 que está há 30 dias sem contato. O scoring equilibra urgência com valor, não deixa o valor engolir tudo.

O P90 do portfólio individual do rep (não a empresa toda) como denominador garante que reps com portfólios de ticket menor ainda consigam pontuar nessa dimensão — sem isso, reps com apenas GTX Basic nunca chegam perto de 20 pts.

D4 distingue Engaging de Prospecting

Deals em Engaging estão em negociação ativa — já houve contato, já há intenção de compra. A diferença de 20 para 10 pontos é deliberada: não queremos punir Prospecting, mas queremos que Engaging sempre saia na frente em paridade de outros fatores.

D5 é um bônus de desempate (10%)

O benchmark contra a equipe serve como tiebreaker secundário e como sinal de alerta. Um deal 7+ dias acima da média do escritório está em território de risco real. 10 pontos máximos garantem que ele apareça no topo sem distorcer o ranking quando D1–D4 já determinam a ordem.

Proxy de contato via engage_date

O dataset original não tem tabela de interações — ela precisa ser construída em produção. No MVP, engage_date é usado como proxy para "data do último contato". Isso subestima D1 para deals com contato recente e superestima para deals sem contato após o engajamento. A aplicação informa o usuário com aviso visual.

Os pesos são configuráveis. A constante CONTACT_THRESHOLD_DAYS = 14 e os valores de D4 podem ser ajustados por produto ou por empresa. O documento SCORING_ENGINE.md especifica como recalibrar com base em dados de execução reais após 90 dias de uso.

Regras de Tiebreaker

Quando dois deals têm o mesmo score numérico, a ordem é determinada sequencialmente:

1
Maior valor de deal
Deal de maior est_value sobe na lista.
2
Engage date mais antiga
Deal em aberto há mais tempo tem prioridade se o valor também for igual.
3
Ordem alfabética por conta
Determinístico — garante que a lista não mude entre recarregamentos sem motivo.
02 — Scoring · 2.6
Explainability
Por que cada deal está na lista — visível para o vendedor em todo item.

Todo item de prioridade exibe um context_reason inline — uma string de até 60 caracteres que identifica o principal motivo pelo qual aquele deal foi priorizado. Sem isso, o vendedor não confia na recomendação.

// Template: "{primary_driver} · {stage} · {value}"

"14 dias sem contato · Engaging · $5.4K"
"Acima da média da equipe em 8 dias · Engaging · $26K"
"Maior valor no portfólio · Prospecting · $3.4K"
"Engaging em andamento · Engaging · $4.8K"exemplos

O driver principal é identificado automaticamente: a dimensão com maior contribuição individual ao score final determina o texto exibido. O vendedor também pode expandir para ver o breakdown completo (D1 até D5).

Critério de validação: a taxa de "Ignorar" em itens recomendados deve ficar abaixo de 20%. Se ultrapassar, o context_reason não está convencendo — revisar a seleção do driver principal.

03 — Limitações · 3.2
Limitações
dos Dados
O que o dataset atual não permite — e como isso afeta a precisão do scoring.

D1 usa proxy, não dado real

A dimensão de maior peso do scoring (25 pontos) mede "tempo sem contato". O dataset não tem tabela de interações. O MVP usa engage_date como proxy — o que significa que D1 sempre sobe com o tempo, nunca zera quando o vendedor faz contato.

Impacto prático: deals com contato recente têm score artificialmente mais alto que o correto. A aplicação exibe um aviso inline informando o usuário. Em produção, isso é o primeiro problema a resolver.

Win rate por vendedor não alimenta o scoring

O histórico de win rate por rep (disponível nos dados) não está integrado ao score de prioridade no MVP. Em produção, um rep com 70% de WR em GTX Plus Pro poderia receber boost nesse produto — aumentando a precisão da priorização para o indivíduo.

Dados históricos fixos (2016–2017)

O dataset cobre um período de 14 meses finalizando em 27/12/2017. Os deals "abertos" no dataset já têm resultado em produção — são abertos apenas para fins de demonstração. A data de referência está fixada em 2017-12-27.

16,2% das linhas sem conta identificada

1.425 oportunidades não têm conta associada. O enriquecimento com setor, porte e subsidiária (de accounts.csv) não é possível para essas linhas. O score não é afetado — account não entra na fórmula — mas a tabela de top contas mostra "Conta não identificada" para esses casos.

5 vendedores sem nenhum deal

Mei-Mei Johns, Elizabeth Anderson, Natalya Ivanova, Carol Thompson e Carl Lin constam em sales_teams.csv mas não têm nenhum deal em sales_pipeline.csv. A aplicação exibe estado vazio para esses reps, sem erro.

Calibração dos pesos ao longo do tempo

Os pesos atuais (D1=25, D2=25, D3=20, D4=20, D5=10) foram definidos por lógica de negócio, não por dados de comportamento. Em produção, é necessário monitorar:

1
Taxa de execução por faixa de score
Itens com score 85–100 devem ter taxa de "Executado" significativamente maior que itens com score 0–54. Se não houver diferença, os pesos não estão discriminando bem.
2
Win rate de deals que aparecem no Top 5
Se o scoring está funcionando, deals no Top 5 devem ter win rate maior que deals fora da lista. Comparar após 90 dias de uso real.
3
Taxa de "Ignorar"
Deve ficar abaixo de 20%. Se o vendedor ignora mais de 1 em cada 5 recomendações, o scoring ou o context_reason não está convencendo.

Recalibrar a cada 90 dias ou quando a taxa de execução cair mais de 10% em relação ao baseline das primeiras 4 semanas. O scoring é um modelo vivo, não uma fórmula fixa.

04 — Submissão · 4.1
Executive
Summary
O que foi construído, como foi construído e por que importa.

Times de vendas perdem negócios não por falta de oportunidades, mas por falta de priorização. O Pipeline Coach AI resolve isso com um scoring engine de 5 dimensões que entrega ao vendedor, todo dia às 08h, exatamente os 5 deals que merecem atenção — com o motivo explícito para cada um.

— Premissa central do produto

Este desafio pediu uma solução funcional para um problema real de RevOps: vendedores priorizando pipeline "no feeling", deixando deals bons esfriarem enquanto perdem tempo em deals que não vão fechar. A solução entregue é uma aplicação web completa — não um script ou um notebook — que qualquer vendedor pode abrir, carregar os CSVs do seu CRM e imediatamente ter uma lista priorizada de ações.

A abordagem foi construída em camadas: primeiro entender os dados a fundo (análise exploratória Python sobre as 8.800 oportunidades), depois especificar o produto com rigor de PRD, depois gerar os artefatos de contexto para agentes de IA, e por fim usar esses artefatos para construir e iterar a aplicação em construtores de site com IA (Lovable e Antigravity). Cada camada alimentou a próxima — o entendimento dos dados informou o scoring; o scoring informou o PRD; o PRD informou os prompts para os agentes construtores.

O resultado: uma aplicação funcional disponível em pipelinecoachai.lovable.app com dashboard de atividades, dashboard de resultados individuais e visão gerencial — tudo rodando no browser, sem backend, alimentado pelos CSVs do dataset.

04 — Submissão · 4.5
Iterações e
Correções
Onde a IA errou, o que estava incompleto, e como foi corrigido.
🔢
Scoring sem data de referência fixa
Claude
Erro da IA
Na primeira versão do scoring engine, o código usava new Date() para calcular "dias em aberto". Como o dataset é de 2017, isso produzia scores absurdos — todos os deals com centenas de dias de aging.
Correção aplicada
Adicionada constante REFERENCE_DATE = new Date('2017-12-27') — última data do dataset. Toda a documentação subsequente foi atualizada para mencionar esse ponto explicitamente e proibir o uso de new Date() em modo de análise.
🔗
Join de produtos quebrando silenciosamente
Claude
Erro da IA
O código inicial fazia join direto de pipeline.product com products.product sem normalizar. "GTX Pro" vs "GTXPro" — o join retornava nulo para todos os deals desse produto, silenciosamente zerando o valor estimado de 729 deals (16,6% do total de Won).
Correção aplicada
Função normalizeProduct() adicionada. O problema foi catalogado em DATA_SCHEMA.md como "Known Data Issue" e em AGENT_INSTRUCTIONS.md como erro proibido — qualquer agente futuro recebe esse alerta antes de escrever código.
💣
parseFloat() quebrando em deals abertos
Claude
Erro da IA
Tentativa de parseFloat(r['close_value']) direto em todos os deals. Para os 2.089 deals abertos, close_value é string vazia "" — não null, não zero. Resultado: NaN propagado por todos os cálculos de valor.
Correção aplicada
Função safeFloat() com verificação de string vazia. Adicionada ao AGENT_INSTRUCTIONS.md como padrão obrigatório. O mesmo comportamento vale para close_date em deals abertos.
📊
Dashboard de resultados ausente — lacuna identificada em uso real
humano
Lacuna original
O MVP gerado pelo primeiro prompt entregava apenas atividades a fazer. O vendedor não tinha nenhuma visão do que já havia realizado — receita, ciclos, produtos vendidos, evolução mês a mês. Isso não estava no briefing original.
Correção aplicada
Nova análise Python dos dados de fechamento, identificação dos 7 gráficos mais relevantes, e criação do MVP_UPDATE_RESULTS_PROMPT.md. A rota /rep/results foi adicionada como segunda iteração do agente construtor.
⚡
Win rate oscilante — risco de ser interpretado como bug
Claude
Risco identificado
O dataset apresenta padrão de ondas pronunciado no win rate mensal (Mar=82%, Abr=48%, Jun=82%, Jul=49%...). Um agente construtor sem esse contexto poderia tentar "corrigir" os dados ou gerar alertas de anomalia desnecessários.
Documentação preventiva
O padrão foi explicado no MVP_UPDATE_RESULTS_PROMPT.md com a nota "É um padrão de pipeline onde deals abertos em um mês fecham em bloco dois meses depois. Não é bug." A documentação antecipou a confusão.
04 — Submissão · 4.6
O Que Só
o Humano Fez
Decisões e contribuições que a IA não faria autonomamente — o julgamento que diferencia a submissão.
Estratégia
Mudar de ChatGPT para Claude no meio do processo
Reconhecer que a decomposição inicial estava feita e que o trabalho de desenvolvimento denso precisava de outra ferramenta. Isso não é óbvio — exige entender os trade-offs de cada modelo.
Diagnóstico
Identificar o gap do dashboard de resultados
O briefing original pedia priorização de atividades. A percepção de que um vendedor também precisa ver o que já fez — não foi a IA que notou isso. Foi o uso real da aplicação entregando essa lacuna.
Curadoria
Escolher quais métricas verificar nos dados
Claude executa as análises, mas quem define "quero ver win rate por mês, distribuição de ciclos por faixa e top contas por receita" é o humano — o julgamento de relevância para o problema de negócio.
Interpretação
Interpretar o padrão de ondas mensais
Claude identificou os números (82%, 48%, 82%, 49%...). Interpretar que é um padrão de waves de pipeline — não ruído nem bug — e documentar isso preventivamente foi julgamento humano aplicado a dado de negócio.
Priorização
Definir os pesos do scoring como decisão de negócio
D1+D2 somam 50% porque "deals bons esfriando" é o problema central — não porque é matematicamente ótimo. Essa calibração inicial é escolha de produto, não de algoritmo. Claude implementa; o humano decide a tese.
Validação
Testar os prompts em dois construtores diferentes
A decisão de rodar os prompts no Lovable E no Antigravity para validar que o resultado independia da plataforma foi metodológica — garante que a spec funciona, não que uma plataforma específica a interpreta bem.
Governança
Criar as regras de copy proibidas
"Nunca escrever 'você está pior que a equipe'" é uma restrição comportamental baseada em risco político real de produtos SaaS de vendas. Claude documenta; o humano define o que não pode acontecer e por quê.
Comunicação
Verificar aderência ao guia de submissão
A pergunta "a documentação possui os requisitos de submissão?" — incluindo o reconhecimento de que o ChatGPT foi usado no início — foi uma auto-auditoria humana que identificou o gap do process log antes da entrega.

A IA executa com velocidade e consistência. O humano direciona com julgamento e contexto de negócio. A qualidade da submissão depende de ambos — não de um ou do outro.
